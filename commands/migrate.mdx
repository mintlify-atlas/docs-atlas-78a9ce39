`/gsd migrate` converts projects from the original GSD (v1) `.planning/` format to GSD-2's `.gsd/` format.

## Command

<ParamField path="/gsd migrate" type="command">
  Migrate `.planning/` directory in current working directory
</ParamField>

<ParamField path="/gsd migrate <path>" type="command">
  Migrate `.planning/` directory at specified path
</ParamField>

## What Gets Converted

The migration tool reads your v1 project structure and maps it to v2:

| v1 Artifact | v2 Artifact | Notes |
|-------------|-------------|-------|
| `PROJECT.md` | `.gsd/PROJECT.md` | Living project description |
| `ROADMAP.md` | `.gsd/M001-ROADMAP.md` | Milestone-scoped roadmaps |
| `REQUIREMENTS.md` | `.gsd/REQUIREMENTS.md` | Requirement states preserved |
| `phases/<phase>/` | `.gsd/milestones/M001/slices/S0X/` | Phases become slices |
| `phases/<phase>/plan.md` | `.gsd/.../S0X-PLAN.md` | Slice plans with task breakdown |
| `phases/<phase>/summary.md` | `.gsd/.../S0X-SUMMARY.md` | Completion summaries |
| `research/` | `.gsd/milestones/M001/M001-RESEARCH.md` | Consolidated research files |
| Decisions (in various files) | `.gsd/DECISIONS.md` | Centralized decision register |


## Basic Usage

### Migrate Current Directory

```bash
cd ~/projects/my-old-project
gsd
/gsd migrate
```

GSD looks for `.planning/` in the current working directory.

### Migrate Specific Path

```bash
gsd
/gsd migrate ~/projects/my-old-project
```

GSD resolves the path and looks for `.planning/` inside it.

### Tilde Expansion

```bash
/gsd migrate ~/projects/payment-gateway
# Expands to /Users/you/projects/payment-gateway/.planning
```

## Validation

Before migrating, GSD validates the source directory:

### Fatal Issues (Block Migration)

- **Missing `.planning/` directory** — Not a v1 project
- **No `PROJECT.md` or `ROADMAP.md`** — Required files missing
- **Malformed phase numbers** — Phases must follow `01-`, `02-`, etc. or `1.0`, `1.1` naming

### Warnings (Migration Proceeds)

- **Missing `requirements.md`** — Defaults to empty requirements
- **Phase without plan** — Phase directory exists but no `plan.md` (treated as pending)
- **Phase with summary but no plan** — Infers completion from summary
- **Duplicate phase numbers** — If milestones use `<details>` blocks, duplicate phase numbers across milestones are allowed

## Supported v1 Variants

The migration tool handles multiple v1 formats:

### Phase Numbering

- **Decimal:** `1.0-setup`, `1.1-database`, `2.0-api`
- **Zero-padded:** `01-setup`, `02-database`, `03-api`
- **Milestone-scoped:** `01-setup` (in M001), `01-setup` (in M002) — different slices

### Roadmap Formats

- **Flat list:** All phases in one milestone
- **Milestone-sectioned:** Roadmap with `## Milestone 1`, `## Milestone 2` headers
- **Details blocks:** Roadmap using `<details><summary>Milestone X</summary>...</details>` for collapsible sections

### Completion Markers

- **`[x]` checkboxes** — Standard markdown checkboxes in roadmap or plan
- **Bold phase names** — `**Phase 1.0: Setup**` (treated as complete)
- **Presence of summary** — If `summary.md` exists, phase is done

If `.gsd/` already exists in the target directory:

```

```

The migration **replaces** the entire `.gsd/` directory. Back up manually if needed.

## Agent Review (Optional)

After writing, GSD offers an **agent-driven review**:

```
✓ Migration complete.

Written to: ~/projects/payment-gateway/.gsd

  [Run agent review]  [Done]
```

The review:

1. Audits the output for GSD-2 standards compliance
2. Checks file structure, frontmatter, task decomposition
3. Identifies missing must-haves, incomplete summaries, or malformed plans
4. Suggests fixes (but does not auto-fix)

**When to use review:**

- Complex v1 projects with many phases
- Custom v1 formats (non-standard naming, unusual structure)
- Before running `/gsd auto` on the migrated project

**When to skip:**

- Simple projects (1 milestone, few slices)
- You plan to inspect manually
- You're iterating on migration (testing multiple times)

## Completion State Preservation

The migration **preserves completion state**:

- `[x]` phases in v1 roadmap → `[x]` slices in v2 roadmap
- Phase with `summary.md` → Slice with `S0X-SUMMARY.md` (copied verbatim)
- Tasks in v1 plan marked `[x]` → Tasks in v2 plan marked `[x]`

You can continue from where v1 left off.

## Research Consolidation

v1 research files are scattered:

```
.planning/
  research/
    codebase-analysis.md
    library-research.md
    api-exploration.md
```

v2 consolidates them into **one research file per milestone**:

```
.gsd/milestones/M001/M001-RESEARCH.md
```

All research content is merged, with headers preserved.

## Decision Extraction

v1 stores decisions in multiple places:

- `PROJECT.md` (architectural notes)
- Phase summaries ("we decided to use X")
- Inline comments in plans

v2 extracts and consolidates them into **`.gsd/DECISIONS.md`** with timestamps.

## Task Decomposition

v1 phases often lack explicit task breakdown. v2 requires tasks.

**If v1 plan has subtasks:**

```markdown
## Phase 1.0: Setup

- [x] Initialize project structure
- [x] Configure database
- [ ] Set up CI/CD
```

**v2 converts to:**

```markdown
# S01: Setup

## Tasks

- [x] T01: Initialize project structure
- [x] T02: Configure database
- [ ] T03: Set up CI/CD
```

**If v1 plan has no subtasks:**

The migration creates **one task per phase** (you can decompose further after migrating).

## Multi-Milestone Projects

If your v1 `ROADMAP.md` has milestone sections:

```markdown
## Milestone 1: Payment Integration

- [x] 1.0: Stripe integration
- [x] 1.1: PayPal integration
- [ ] 1.2: Webhook handling

## Milestone 2: User Management

- [ ] 2.0: User signup
- [ ] 2.1: User login
```

**v2 creates:**

```
.gsd/milestones/
  M001-PAYMENT-INTEGRATION/
    M001-ROADMAP.md
    slices/
      S01-stripe-integration/
      S02-paypal-integration/
      S03-webhook-handling/
  M002-USER-MANAGEMENT/
    M002-ROADMAP.md
    slices/
      S01-user-signup/
      S02-user-login/
```

## Common Issues

### Issue: "Directory not found"

**Problem:** No `.planning/` directory at the specified path.

**Fix:**

```bash
# Check if .planning exists
ls -la ~/projects/my-project/.planning

# If missing, you're not migrating a v1 project
# Use /gsd for new projects instead
```

### Issue: "Fatal: Missing PROJECT.md"

**Problem:** The `.planning/` directory exists but lacks core files.

**Fix:**

Ensure your v1 project has:

```
.planning/
  PROJECT.md       ← Required
  ROADMAP.md       ← Required
  requirements.md  ← Optional
  phases/          ← Optional (can be empty)
```

### Issue: "Phase numbers not sequential"

**Problem:** Phase numbering like `01-`, `03-`, `05-` (missing 02, 04).

**Fix:**

Renumber phases to be sequential, or ignore the warning (migration proceeds anyway).

### Issue: "Duplicate phase numbers across milestones"

**Problem:** Phase `01-` appears in both Milestone 1 and Milestone 2.

**Fix:**

This is allowed if your roadmap uses `<details>` blocks or milestone headers. The migration scopes phases to their milestone.

## After Migration

### Validate the Output

Run GSD doctor to check integrity:

```bash
/gsd doctor
```

If issues are found, fix them or run:

```bash
/gsd doctor fix
```

### Resume Work

Pick up where you left off:

```bash
/gsd auto
# Continues from the first incomplete slice
```

### Compare with v1

Inspect the differences:

```bash
# Old structure
ls .planning/

# New structure
ls .gsd/

# Compare specific files
diff .planning/phases/01-setup/plan.md .gsd/milestones/M001/slices/S01-setup/S01-PLAN.md
```
